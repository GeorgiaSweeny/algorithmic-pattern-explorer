# User Stories — Derived from Educator Stakeholder Consultation

This document translates the findings from [inital-educator-stakeholder-consulation-summary.md](inital-educator-stakeholder-consulation-summary.md) into user stories, grouped by theme and tagged with MoSCoW priority (Must / Should / Could / Future) to reflect the MVP framing already established in that summary.

Format: **As a** `<role>`, **I want** `<goal>`, **so that** `<benefit>`.

---

## 1. Target Audience & Learning Levels

**US-1.1 (Must)**
As a learner with no prior programming experience, I want the interface and explanations to be understandable without technical background, so that I can engage with the system regardless of age or prior knowledge.

**US-1.2 (Should)**
As an educator teaching different age groups (primary through adult), I want the system to adapt its explanations to different levels of prior knowledge, so that I can use the same tool across multiple classes or learner groups.

**US-1.3 (Could)**
As a self-directed adult learner, I want to choose a more technical/mathematical explanation depth, so that I can get value from the system beyond a beginner level.

---

## 2. Tool vs Curriculum Positioning

**US-2.1 (Must)**
As a learner, I want to explore algorithms independently without needing a teacher present, so that I can use the system for self-directed learning.

**US-2.2 (Future)**
As an educator, I want a structured lesson/curriculum mode, so that I can lead a class through a defined sequence of learning activities rather than free exploration.

**US-2.3 (Future)**
As an educator, I want educator-led session controls, so that I can guide a group of learners through the same content in sync.

---

## 3. Layered / Multi-Level Explanation

**US-3.1 (Must)**
As a beginner learner, I want node documentation written in plain language, so that I can understand what a node does without technical jargon.

**US-3.2 (Should)**
As an intermediate learner, I want a conceptual explanation of each node, so that I can understand the underlying computational idea, not just the surface effect.

**US-3.3 (Could)**
As an advanced learner, I want access to technical or mathematical detail for each node, so that I can understand the precise algorithmic/mathematical basis of its behaviour.

---

## 4. Multi-Modal Learning

**US-4.1 (Must)**
As a learner, I want each node to show its behaviour visually as I interact with it, so that I can connect cause and effect directly.

**US-4.2 (Must)**
As a learner, I want to adjust parameters on a node and see the output update, so that I can learn through direct experimentation.

**US-4.3 (Should)**
As a learner, I want an optional animation showing a node's behaviour over time or across steps, so that I can reinforce understanding through repetition and visual pacing.

**US-4.4 (Must)**
As a learner, I want a written explanation alongside the visual and interactive elements, so that I have multiple channels reinforcing the same concept.

---

## 5. Guided Onboarding

**US-5.1 (Should)**
As a first-time user, I want a guided tutorial when I open the system, so that I understand how to use the node graph, canvas, and documentation panel before exploring on my own.

**US-5.2 (Should)**
As a learner, I want tooltips and hover explanations on interface elements, so that I can get help without leaving my current task.

**US-5.3 (Could)**
As a learner, I want contextual prompts suggesting what to try next, so that I don't feel lost when first exploring a new algorithm.

---

## 6. Clear Learning Outcomes

**US-6.1 (Must)**
As a learner, I want to see the explicit learning objective for a node or algorithm, so that I understand why I'm engaging with it.

**US-6.2 (Should)**
As a learner or educator, I want nodes mapped to computational thinking concepts (e.g. decomposition, abstraction, iteration), so that the educational purpose of each interaction is clear.

**US-6.3 (Should)**
As a learner, I want to see how a node/algorithm fits into an overall progression, so that I understand what I've learned and what comes next.

---

## 7. Engagement & Motivation

**US-7.1 (Future)**
As a learner, I want challenges or achievements tied to my exploration, so that I stay motivated to continue learning.

**US-7.2 (Future)**
As a learner, I want to unlock new content or nodes as I progress, so that there is a sense of advancement.

**US-7.3 (Future)**
As an educator, I want a progression/gamification system I can point learners toward, so that engagement is sustained outside guided sessions.

---

## 8. Assessment

**US-8.1 (Future)**
As a learner, I want to be shown a node's animation and asked to identify its function, so that I can test my understanding.

**US-8.2 (Future)**
As a learner, I want to rebuild a target pattern using available nodes, so that I can demonstrate applied understanding.

**US-8.3 (Future)**
As a learner, I want to debug an incorrect node graph, so that I can practise diagnostic and computational thinking skills.

**US-8.4 (Future)**
As a learner, I want to match a node graph to its resulting output, so that I can practise reading structure-to-behaviour relationships.

**US-8.5 (Future)**
As an educator, I want an assessment engine driven by node metadata, so that I can generate or evaluate exercises without manually authoring them.

---

## 9. Cognitive Load & Clarity

**US-9.1 (Must)**
As a new learner, I want a minimal initial interface with information revealed progressively, so that I am not overwhelmed when first using the system.

**US-9.2 (Must)**
As a learner, I want information chunked into small, digestible sections, so that I can process one concept at a time.

**US-9.3 (Should)**
As a learner, I want contextual help available only when I need it, so that the interface stays uncluttered during normal use.

---

## 10. Progression of Difficulty

**US-10.1 (Must)**
As a learner, I want algorithms organised into a tiered structure (core, intermediate, advanced), so that I can progress from simple to complex concepts gradually.

**US-10.2 (Should)**
As a learner, I want new concepts introduced incrementally rather than all at once, so that my learning builds on what I already know.

---

## 11. Concept Linking Across Algorithms

**US-11.1 (Should)**
As a learner, I want to recognise shared concepts (e.g. rotation, repetition, randomness) as they reappear across different algorithms, so that I build transferable understanding rather than isolated facts.

**US-11.2 (Should)**
As a learner, I want the system to highlight when a concept I've already learned reappears in a new context, so that I can connect prior learning to new material.

**US-11.3 (Could)**
As a learner, I want a comparison view across algorithms that share a concept, so that I can see the concept applied in different settings side by side.

---

## 12. System Scope & Modes

**US-12.1 (Could)**
As a curious learner, I want the option to view underlying code for a node/algorithm, so that I can go deeper if I'm interested.

**US-12.2 (Future)**
As a learner, I want to create and save my own patterns, so that I can express creativity beyond guided exploration (Sandbox Mode).

**US-12.3 (Future)**
As a learner, I want to build custom node graphs from scratch, so that I can construct my own algorithms (Builder Mode).

**US-12.4 (Future)**
As a learner, I want to save and share the patterns I create, so that I can revisit or show my work to others.

**US-12.5 (Must)**
As a learner, I want an Explorer Mode where I step through existing algorithms without needing to build anything, so that I have a low-barrier entry point into the system.
